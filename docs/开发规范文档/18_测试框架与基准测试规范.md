# 18. 测试框架与基准测试规范

## 18.1 测试框架概述

幻语内置完整的单元测试、集成测试和基准测试框架，支持测试的自动发现、并行执行和详细的报告输出。测试框架深度集成于编译器和工具链中，通过 `huan test` 命令运行。

**核心特性**：
- **声明式测试定义**：使用 `测试.模块` 和 `测试.测试` 组织测试用例
- **丰富的断言 API**：提供多种断言方法，失败时输出详细信息
- **测试隔离**：每个测试在独立的任务中运行，互不干扰
- **并行执行**：默认利用多核并行运行测试，显著缩短测试时间
- **基准测试**：支持性能基准测试，可检测性能回归
- **覆盖率报告**：生成代码覆盖率报告（需 `--coverage` 标志）
- **测试过滤**：支持按名称模式筛选要运行的测试

## 18.2 测试模块与测试用例定义

### 18.2.1 测试模块语法

```hl
导入 幻语.测试 为 测试

测试.模块("模块名称", || {
    测试.测试("测试名称", || {
        -- 测试代码
    })
    
    测试.测试("另一个测试", || {
        -- 测试代码
    })
})
```

**嵌套模块**：
```hl
测试.模块("外层模块", || {
    测试.模块("内层模块", || {
        测试.测试("内层测试", || { ... })
    })
})
```

### 18.2.2 测试属性

可使用属性控制测试行为：

```hl
@测试.忽略("原因说明")     -- 跳过此测试
@测试.仅运行              -- 只运行标记的测试（需 --ignored 标志）
@测试.顺序                -- 强制串行执行
@测试.超时(秒数)          -- 设置测试超时时间
```

**示例**：
```hl
测试.模块("网络测试", || {
    @测试.忽略("需要网络连接")
    @测试.超时(10)
    测试.测试("下载文件", || {
        令 响应 = HTTP.获取("https://example.com")
        测试.断言相等(响应.状态码, 200)
    })
})
```

### 18.2.3 测试夹具（Fixture）

使用 `测试.前置` 和 `测试.后置` 定义测试前后的准备和清理工作：

```hl
测试.模块("数据库测试", || {
    令 连接 = 空
    
    测试.前置(|| {
        连接 = 数据库.连接("test_db")
        连接.清空所有表()
    })
    
    测试.后置(|| {
        连接.关闭()
    })
    
    测试.测试("插入数据", || {
        连接.执行("INSERT INTO users VALUES (1, '张三')")
        令 结果 = 连接.查询("SELECT * FROM users")
        测试.断言相等(结果.行数(), 1)
    })
})
```

## 18.3 断言 API

### 18.3.1 基本断言

| 断言函数 | 说明 |
|---------|------|
| `测试.断言(条件: 布尔)` | 断言条件为真 |
| `测试.断言相等(实际: T, 期望: T)` | 断言两个值相等（需要 `T: 可比较`） |
| `测试.断言不等(实际: T, 期望: T)` | 断言两个值不相等 |
| `测试.断言为真(值: 布尔)` | 断言值为真 |
| `测试.断言为假(值: 布尔)` | 断言值为假 |
| `测试.断言为空(值: 可选<T>)` | 断言可选值为空 |
| `测试.断言不为空(值: 可选<T>)` | 断言可选值不为空 |
| `测试.断言成功(结果: 结果<T, E>)` | 断言结果为成功 |
| `测试.断言错误(结果: 结果<T, E>)` | 断言结果为错误 |

### 18.3.2 数值断言

| 断言函数 | 说明 |
|---------|------|
| `测试.断言近似(实际: 浮点64, 期望: 浮点64, 误差: 浮点64)` | 断言浮点数在误差范围内相等 |
| `测试.断言大于(甲: T, 乙: T)` | 断言甲 > 乙 |
| `测试.断言小于(甲: T, 乙: T)` | 断言甲 < 乙 |
| `测试.断言大于等于(甲: T, 乙: T)` | 断言甲 >= 乙 |
| `测试.断言小于等于(甲: T, 乙: T)` | 断言甲 <= 乙 |

### 18.3.3 字符串断言

| 断言函数 | 说明 |
|---------|------|
| `测试.断言包含(字符串: 字符串, 子串: 字符串)` | 断言字符串包含子串 |
| `测试.断言不包含(字符串: 字符串, 子串: 字符串)` | 断言字符串不包含子串 |
| `测试.断言以...开头(字符串: 字符串, 前缀: 字符串)` | 断言字符串以指定前缀开头 |
| `测试.断言以...结尾(字符串: 字符串, 后缀: 字符串)` | 断言字符串以指定后缀结尾 |
| `测试.断言匹配(字符串: 字符串, 正则表达式: 字符串)` | 断言字符串匹配正则表达式 |

### 18.3.4 集合断言

| 断言函数 | 说明 |
|---------|------|
| `测试.断言列表相等(实际: 列表<T>, 期望: 列表<T>)` | 断言两个列表元素完全相同 |
| `测试.断言包含元素(列表: 列表<T>, 元素: T)` | 断言列表包含指定元素 |
| `测试.断言字典包含键(字典: 字典<K, V>, 键: K)` | 断言字典包含指定键 |

### 18.3.5 异常断言

| 断言函数 | 说明 |
|---------|------|
| `测试.断言触发错误(闭包: 函数())` | 断言闭包执行时触发错误（panic） |
| `测试.断言不触发错误(闭包: 函数())` | 断言闭包执行时不触发错误 |
| `测试.断言触发错误消息(闭包: 函数(), 期望消息: 字符串)` | 断言闭包触发错误且消息包含指定文本 |

### 18.3.6 自定义断言消息

所有断言函数都支持可选的最后一个参数作为自定义失败消息：

```hl
测试.断言相等(实际, 期望, "计算错误：{} ≠ {}", 实际, 期望)
```

### 18.3.7 使用示例

```hl
测试.模块("断言示例", || {
    测试.测试("基本断言", || {
        令 甲 = 42
        令 乙 = 42
        测试.断言相等(甲, 乙)
        测试.断言为真(甲 == 乙)
        测试.断言大于(甲, 40)
    })
    
    测试.测试("浮点断言", || {
        令 圆周率 = 3.14159
        测试.断言近似(圆周率, 3.1416, 0.001)
    })
    
    测试.测试("字符串断言", || {
        令 文本 = "Hello, 世界"
        测试.断言包含(文本, "世界")
        测试.断言以...开头(文本, "Hello")
    })
    
    测试.测试("列表断言", || {
        令 列表 = [1, 2, 3]
        测试.断言列表相等(列表, [1, 2, 3])
        测试.断言包含元素(列表, 2)
    })
    
    测试.测试("异常断言", || {
        测试.断言触发错误(|| {
            终止(1)
        })
        
        测试.断言触发错误消息(|| {
            令 结果 = 10 / 0
        }, "除零")
    })
})
```

## 18.4 基准测试

### 18.4.1 基准测试定义

使用 `测试.基准` 定义性能基准测试：

```hl
测试.模块("性能基准", || {
    测试.基准("斐波那契数列", || {
        令 结果 = 斐波那契(40)
        测试.防止优化(结果)
    })
    
    测试.基准("列表追加", || {
        令 列表 = 列表<整数>.新建()
        对于 索引 在 0..1000 中
            列表.追加(索引)
        结束
        测试.防止优化(列表)
    })
})
```

### 18.4.2 防止优化

`测试.防止优化(值)` 函数用于阻止编译器优化掉被测试的代码，确保基准测试的准确性。

### 18.4.3 基准测试选项

基准测试支持以下属性：

```hl
@测试.基准.最小样本数(100)     -- 最少采样次数
@测试.基准.最小时间(秒数)        -- 最少运行时间
@测试.基准.最大时间(秒数)        -- 最多运行时间
@测试.基准.预热迭代(10)         -- 预热迭代次数
```

**示例**：
```hl
@测试.基准.最小时间(5)           -- 至少运行 5 秒
@测试.基准.最小样本数(1000)      -- 至少 1000 次采样
测试.基准("复杂计算", || {
    令 结果 = 复杂算法(1000)
    测试.防止优化(结果)
})
```

### 18.4.4 基准测试输出

基准测试运行后输出详细的统计信息：

```
基准测试: 斐波那契数列
  样本数: 1000
  平均时间: 15.234 ms
  标准差: 0.512 ms
  中位数: 15.102 ms
  最小值: 14.876 ms
  最大值: 17.234 ms
  吞吐量: 65.64 ops/s
```

### 18.4.5 性能回归检测

使用 `测试.基准.比较` 与历史基线比较：

```hl
测试.基准.比较("排序性能", "基线_排序.json", || {
    令 数据 = 生成随机数据(10000)
    数据.排序()
    测试.防止优化(数据)
})
```

若性能下降超过阈值（默认 5%），基准测试将标记为失败。

### 18.5.3 测试运行器实现

```rust
pub struct TestRunner {
    config: TestConfig,
    tests: Vec<Test>,
    results: Vec<TestResult>,
}

pub struct TestConfig {
    pub filter: Option<String>,
    pub exact: bool,
    pub nocapture: bool,
    pub jobs: usize,
    pub include_ignored: bool,
    pub only_ignored: bool,
    pub fail_fast: bool,
    pub format: OutputFormat,
    pub report_time: bool,
}

impl TestRunner {
    pub fn new(config: TestConfig) -> Self;
    
    /// 发现并收集所有测试
    pub fn discover(&mut self, paths: &[PathBuf]) -> Result<usize>;
    
    /// 运行所有测试
    pub fn run(&mut self) -> TestSummary;
    
    /// 运行单个测试
    fn run_test(&self, test: &Test) -> TestResult;
}

#[derive(Debug)]
pub struct Test {
    pub name: String,
    pub module_path: String,
    pub location: SourceLocation,
    pub ignored: bool,
    pub ignore_reason: Option<String>,
    pub timeout: Option<Duration>,
    pub serial: bool,
    pub func: Box<dyn FnOnce() + Send>,
}

#[derive(Debug)]
pub struct TestResult {
    pub test: Test,
    pub status: TestStatus,
    pub duration: Duration,
    pub output: Option<String>,
}

#[derive(Debug)]
pub enum TestStatus {
    Passed,
    Failed { message: String, location: SourceLocation },
    Ignored { reason: Option<String> },
    TimedOut,
}
```

### 18.5.4 并行执行策略

测试运行器使用线程池并行执行测试：

1. **依赖分析**：识别需要串行执行的测试（标记 `@测试.顺序`）
2. **任务调度**：将测试分配给工作线程
3. **输出同步**：收集各线程的输出，按测试顺序整理
4. **资源隔离**：每个测试在独立的任务中运行，避免状态污染

```rust
impl TestRunner {
    fn run_parallel(&mut self, tests: Vec<Test>) -> Vec<TestResult> {
        let (tx, rx) = std::sync::mpsc::channel();
        let pool = ThreadPool::new(self.config.jobs);
        
        for test in tests {
            let tx = tx.clone();
            let config = self.config.clone();
            pool.execute(move || {
                let result = run_single_test(test, config);
                tx.send(result).unwrap();
            });
        }
        
        drop(tx);
        rx.iter().collect()
    }
}
```

## 18.9 测试最佳实践指南

### 18.9.1 测试命名

- 测试模块名应与被测模块名对应
- 测试名应描述测试场景和期望结果
- 使用中文命名提高可读性

### 18.9.2 测试结构

每个测试应遵循 AAA 模式：
1. **Arrange（准备）**：设置测试数据和环境
2. **Act（执行）**：调用被测代码
3. **Assert（断言）**：验证结果

```hl
测试.测试("加法运算", || {
    -- Arrange
    令 甲 = 10
    令 乙 = 20
    
    -- Act
    令 结果 = 甲 + 乙
    
    -- Assert
    测试.断言相等(结果, 30)
})
```

### 18.9.3 避免测试间依赖

- 测试应独立运行，不依赖其他测试的执行顺序
- 使用 `测试.前置` 和 `测试.后置` 确保状态隔离
- 避免使用全局可变状态

### 18.9.4 基准测试注意事项

- 预热迭代足够多以稳定性能
- 使用 `测试.防止优化` 确保代码不被消除
- 避免在基准测试中进行 I/O 操作

## 18.11 与持续集成（CI）集成

### 18.11.1 GitHub Actions 配置

```yaml
name: 幻语 CI

on: [push, pull_request]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      
      - name: 安装幻语
        run: curl -fsSL https://huanlang.org/install.sh | sh
      
      - name: 运行测试
        run: huan test --format junit --coverage
      
      - name: 上传测试报告
        uses: actions/upload-artifact@v4
        with:
          name: test-results
          path: |
            target/junit.xml
            target/coverage/
```
